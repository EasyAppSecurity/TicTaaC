Threat Modeling Report

Name: Threat Model Name, version: 1.0.1, date: 2022-05-09 10:59

The purpose of this document is to represent the results of the threat modeling exercise, analyze risks for identified threats and provide the understanding of the mitigation controls needed for the software system.

Performing threat model at the design level helps in identification of vulnerabilities that need to be closed at the design level and input this into the build phase, identification of information assets that need security controls, mapping of identified security controls into technical/administrative/physical controls, identification of security test cases/security test scenarios to test the security requirements.

The threat modeling process should be driven by application designers, however, developers and testers should be involved.

Results of this threat modeling exercise are prioritized by risk level list of identified threats and list of proposed mitigations for those threats.

 

Threat Analysis

                               

The following methodologies are used for threats classification during the assessment:

  • Microsoft STRIDE Methodology is used to identify threats relevant to the application. STRIDE was developed to educate developers on how to think about computer security threats.  For a detailed description of STRIDE threats please visit Microsoft STRIDE page.

  • The OWASP Top 10 is a standard awareness methodology for developers and web application security. It represents a broad consensus about the most critical security risks to web applications.  For the addition information please see OWASP Top Ten.
